### 1. 🔴 **P0：Cloudinary 圖片加載阻塞**

#### 問題位置
- [App.tsx](App.tsx#L75) - Header 中的 logo：
  ```tsx
  <img
    src="https://res.cloudinary.com/dvizdsv4m/image/upload/f_auto,q_70,w_300/v1768743629/Dessert-Chinese_u8uoxt.png"

#### 影響數據
- ❌ **首屏停留**：+1.5-2 秒（完全阻塞 First Paint）

#### 優化方案（立即實施）

```

**方案 B：智能圖片加載策略**

### 2. 🔴 **P0：LINE LIFF 初始化阻塞**

#### 問題位置
- [LiffContext.tsx](src/contexts/LiffContext.tsx#L28) - 同步初始化 LIFF
- [index.tsx](index.tsx#L8-14) - 在 App Root 層級同步載入

```tsx
// ❌ 當前做法：同步初始化，阻塞渲染
useEffect(() => {
  liff.init({ liffId })  // 等待網路，主線程卡住
    .then(() => {
      setLiffObject(liff);
      if (liff.isLoggedIn()) {
        // 再次網路請求獲取 profile
        liff.getProfile().then(...)
      }
    })
}, [liffId]);
```

#### 為什麼慢？
1. **雙重網路請求**：LIFF init + getProfile = 2 個往返 (RTT)
2. **無超時控制**：如果 LINE 伺服器慢，無限期等待
3. **不是 Promise 並行**：getProfile 沒有和其他操作並行

#### 影響數據
- ❌ **初始化延遲**：+1-2 秒（在 liff.init 處卡住）
- ❌ **無 Fallback**：如果 LIFF 失敗，整個 App 凍結

#### 優化方案（立即實施）

**方案 A：非阻塞式初始化 + 快速 Fallback**
```tsx
// src/contexts/LiffContext.tsx
export const LiffProvider: React.FC<{ children: ReactNode }> = ({ children }) => {
  const [liffReady, setLiffReady] = useState(false);

  useEffect(() => {
    const liffId = import.meta.env.VITE_LIFF_ID;
    if (!liffId) return;

    // ⭐ 關鍵：將 LIFF init 改為 non-blocking，設置超時
    const initLiff = async () => {
      try {
        // 1. 超時控制（5秒後自動 fallback）
        const liffInit = Promise.race([
          liff.init({ liffId }),
          new Promise((_, reject) => 
            setTimeout(() => reject(new Error('LIFF timeout')), 5000)
          )
        ]);

        await liffInit;

        // 2. 並行獲取 profile 和其他資料
        if (liff.isLoggedIn()) {
          const [profile, context] = await Promise.all([
            liff.getProfile(),
            liff.getContext()
          ]);
          setProfile({ ...profile });
        }
      } catch (err) {
        // ✅ 優雅 Fallback：沒有 LIFF 也能使用 App
        console.warn('LIFF initialization failed, using localStorage fallback');
        setIsLoggedIn(false);
      }

      setLiffReady(true);  // App 可以繼續渲染
    };

    initLiff();
  }, []);

  return (
    <LiffContext.Provider value={...}>
      {liffReady ? children : <AppSkeleton />}  {/* 只顯示加載狀態，不阻塞渲染 */}
    </LiffContext.Provider>
  );
};
```

**方案 B：檢查 localStorage 快取**
```tsx
// ⭐ 如果用戶已登入過，用快取 profile
useEffect(() => {
  const cachedProfile = localStorage.getItem('liff_profile');
  
  if (cachedProfile) {
    // 快速恢復上次狀態（0ms）
    setProfile(JSON.parse(cachedProfile));
    setIsLoggedIn(true);
  }

  // 後台初始化，不阻塞 UI
  initLiffInBackground();
}, []);
```

- ⏱️ **初始化時間**：-1-1.5 秒（改為非阻塞）

### 3. 🟠 **P1：React 大型組件樹**

#### 問題位置
- [App.tsx](App.tsx#L1-150) - 單一 1263 行超大組件
  - 包含：Header + LandingScreen + PassportScreen + ARScanner + ...
  - 每次狀態更新，所有子組件重新渲染

#### 為什麼慢？
1. **無代碼分割**：所有功能打包在一個 JS 文件中
2. **無 Memo 優化**：子組件無 React.memo，父組件更新時全部重新渲染
3. **邏輯混亂**：Header 監聽 stampCount（1秒輪詢），導致頻繁渲染

#### 影響數據
- ❌ **主線程阻塞**：+200-400ms（每次狀態更新）
- ❌ **重複計算**：getUnlockedStampCount() 每次都被調用

```tsx
// ❌ 問題代碼：App.tsx line 65-70
setInterval(() => {
  setStampCount(getUnlockedStampCount());
}, 1000);  // 每秒強制渲染一次！
```

#### 優化方案（短期實施）

**方案 A：代碼分割 + Lazy Load**
```tsx

```

**方案 B：優化 Header 的 stampCount 監聽**
```tsx
// ❌ 舊做法：每秒輪詢 + 無謂重渲染
const [stampCount, setStampCount] = useState(0);
useEffect(() => {
  const interval = setInterval(() => {
    setStampCount(getUnlockedStampCount());  // 1秒檢查一次！
  }, 1000);
  return () => clearInterval(interval);
}, []);

// ✅ 新做法：事件驅動，只在必要時更新
// src/hooks/useStampCount.ts
export function useStampCount() {
  const [stampCount, setStampCount] = useState(getUnlockedStampCount());

  useEffect(() => {
    // 監聽 localStorage 變化，不是盲目輪詢
    const handleStorageChange = () => {
      setStampCount(getUnlockedStampCount());
    };

    window.addEventListener('storage', handleStorageChange);
    // 自訂事件：當 PassportScreen 解鎖印章時
    document.addEventListener('stamp-unlocked', handleStorageChange);

    return () => {
      window.removeEventListener('storage', handleStorageChange);
      document.removeEventListener('stamp-unlocked', handleStorageChange);
    };
  }, []);

  return stampCount;
}

// App.tsx 使用
const Header = () => {
  const stampCount = useStampCount();  // 只在 localStorage 改變時更新
  return (...)
};
```

### 4. 🟠 **P1：GA4 事件追蹤同步阻塞**

#### 問題位置
- [analytics.ts](analytics.ts#L40-50) - trackEvent 同步調用
- [PassportScreen.tsx](PassportScreen.tsx#L180-220) - 每次互動都追蹤

```tsx
// ❌ 問題：track 調用在主線程
export const trackEvent = (eventName: string, eventParams?: Record<string, any>) => {
  if (window.gtag) {
    window.gtag('event', eventName, withSiteId(eventParams));  // 同步阻塞
  }
};

// ❌ 被調用的地方：每次點擊都會同步追蹤
<button onClick={() => {
  trackButtonClick('unlock_stamp', 'passport');  // 阻塞點擊反應
  unlockStamp(stampId);
}}>
```

#### 為什麼慢？
1. **主線程競爭**：gtag 調用與 React 渲染搶佔同一線程
2. **無錯誤隔離**：GA4 異常會影響主應用
3. **不必要的同步**：分析數據不影響 UX，不需同步

#### 影響數據
- ❌ **點擊延遲**：每次互動 +5-20ms
- ❌ **累積延遲**：10 次互動 = +50-200ms 卡頓

#### 優化方案（短期實施）

**方案 A：非同步 + Web Worker**
```tsx
// src/utils/analytics-worker.ts
if (typeof Worker !== 'undefined') {
  const analyticsWorker = new Worker(new URL('./analytics.worker.ts', import.meta.url), {
    type: 'module'
  });

  export const trackEventAsync = (eventName: string, params?: Record<string, any>) => {
    // 丟給 Worker，不阻塞主線程
    analyticsWorker.postMessage({ eventName, params });
  };
}

// src/utils/analytics.worker.ts
self.onmessage = (event: MessageEvent) => {
  const { eventName, params } = event.data;
  // Worker 線程中調用 gtag，不影響主線程
  if (window.gtag) {
    window.gtag('event', eventName, params);
  }
};
```

**方案 B：requestIdleCallback 延遲追蹤**
```tsx
// ✅ 推薦做法：使用 requestIdleCallback
export const trackEvent = (eventName: string, params?: Record<string, any>) => {
  if ('requestIdleCallback' in window) {
    requestIdleCallback(() => {
      if (window.gtag) {
        window.gtag('event', eventName, params);
      }
    }, { timeout: 2000 });  // 最多延遲 2 秒
  } else {
    // Fallback：setTimeout
    setTimeout(() => {
      if (window.gtag) {
        window.gtag('event', eventName, params);
      }
    }, 0);
  }
};
```

**方案 C：分批推送 + 隊列機制**
```tsx
// src/utils/analyticsQueue.ts
class AnalyticsQueue {
  private queue: Array<{eventName: string, params: any}> = [];
  private isProcessing = false;

  push(eventName: string, params: any) {
    this.queue.push({ eventName, params });
    if (!this.isProcessing) {
      this.processBatch();
    }
  }

  private async processBatch() {
    this.isProcessing = true;

    // 等待網路空閒（100ms）
    while (this.queue.length > 0) {
      const batch = this.queue.splice(0, 5);  // 一次 5 個
      
      requestIdleCallback(() => {
        batch.forEach(({ eventName, params }) => {
          if (window.gtag) {
            window.gtag('event', eventName, params);
          }
        });
      });

      await new Promise(resolve => setTimeout(resolve, 100));
    }

    this.isProcessing = false;
  }
}

export const analyticsQueue = new AnalyticsQueue();
```

#### 預期效果
- ⏱️ **點擊反應延遲**：-5-15ms（改為非同步）
- ✅ **FID（首次輸入延遲）**：100-150ms → 50-80ms
- ✅ **用戶感受**：按鈕點擊更流暢

## 💬 Q&A

**Q1：為什麼不用 CDN 加速 Cloudinary？**
